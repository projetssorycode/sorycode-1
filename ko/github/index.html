GitHub

GitHub issue와 pull request에서 SoryCode를 사용하세요.

SoryCode는 GitHub 워크플로와 통합됩니다. 댓글에 /sorycode 또는 /oc를 mention하면 SoryCode가 GitHub Actions runner 안에서 작업을 실행합니다.

기능

  • Issue triage: SoryCode에게 issue를 분석하고 내용을 설명하도록 요청할 수 있습니다.
  • Fix and implement: SoryCode에게 issue 수정이나 기능 구현을 요청할 수 있습니다. 새 branch에서 작업한 뒤 변경 사항을 담은 PR을 생성합니다.
  • Secure: SoryCode는 GitHub runner 내부에서 실행됩니다.

설치

GitHub repo에 연결된 프로젝트에서 아래 명령을 실행하세요.

이 명령은 GitHub app 설치, workflow 생성, secrets 설정 과정을 안내합니다.

Manual Setup

원하면 수동으로도 설정할 수 있습니다.

  1. Install the GitHub app

    github.com/apps/sorycode-agent로 이동하세요. 대상 repo에 app이 설치되어 있는지 확인하세요.

  2. Add the workflow

    아래 workflow 파일을 repo의 .github/workflows/sorycode.yml에 추가하세요. env에는 필요한 API key를 넣고, model은 환경에 맞게 설정하세요.

  3. Store the API keys in secrets

    조직 또는 프로젝트 Settings에서 왼쪽의 Secrets and variables를 펼친 뒤 Actions를 선택하세요. 필요한 API key를 추가하면 됩니다.

구성

  • model: SoryCode에서 사용할 model입니다. provider/model 형식이며 필수입니다.

  • agent: 사용할 agent입니다. primary agent여야 합니다. 찾지 못하면 config의 default_agent를 사용하고, 그것도 없으면 "build"로 fallback합니다.

  • share: SoryCode 세션 공유 여부입니다. public repo에서는 기본값이 true입니다.

  • prompt: 기본 동작을 override하는 선택형 custom prompt입니다. SoryCode의 요청 처리 방식을 조정할 때 사용합니다.

  • token: 댓글 생성, 커밋, PR 생성 같은 작업을 수행할 때 사용하는 선택형 GitHub access token입니다. 기본적으로 SoryCode는 SoryCode GitHub App의 installation access token을 사용하므로, 커밋/댓글/PR 작성 주체가 app으로 표시됩니다.

    또는 SoryCode GitHub App을 설치하지 않고도 GitHub Action runner의 기본 제공 GITHUB_TOKEN을 사용할 수 있습니다. 이 경우 workflow에 필요한 permission을 반드시 부여하세요.

    필요하면 personal access token(PAT)도 사용할 수 있습니다.

Supported Events

SoryCode는 아래 GitHub event로 트리거할 수 있습니다.

Event TypeTriggered ByDetails
issue_commentissue 또는 PR 댓글댓글에 /sorycode 또는 /oc를 mention하세요. SoryCode가 맥락을 읽고 branch 생성, PR 생성, 답변을 수행할 수 있습니다.
pull_request_review_commentPR의 특정 코드 줄 댓글코드 리뷰 중 /sorycode 또는 /oc를 mention하세요. SoryCode가 파일 경로, 라인 번호, diff 맥락을 받습니다.
issuesissue 생성 또는 수정issue가 생성/수정될 때 SoryCode를 자동 트리거합니다. prompt 입력이 필요합니다.
pull_requestPR 생성 또는 업데이트PR open/synchronize/reopen 시 SoryCode를 자동 트리거합니다. 자동 리뷰에 유용합니다.
schedulecron 기반 스케줄스케줄에 따라 SoryCode를 실행합니다. prompt 입력이 필요합니다. 출력은 로그와 PR로 남습니다(issue 댓글 대상 없음).
workflow_dispatchGitHub UI에서 수동 실행Actions 탭에서 필요 시 SoryCode를 실행합니다. prompt 입력이 필요하며 출력은 로그와 PR로 남습니다.

Schedule Example

자동화 작업을 위해 스케줄 기반으로 SoryCode를 실행할 수 있습니다.

schedule event는 지시를 추출할 댓글이 없기 때문에 prompt 입력이 필수입니다. 또한 schedule workflow는 permission 체크용 사용자 맥락 없이 실행되므로, SoryCode가 branch나 PR을 만들게 하려면 contents: write와 pull-requests: write를 부여해야 합니다.

Pull Request Example

PR이 열리거나 업데이트될 때 자동 리뷰를 수행할 수 있습니다.

pull_request event에서 prompt를 지정하지 않으면 SoryCode는 pull request 리뷰를 기본 동작으로 수행합니다.

Issues Triage Example

새로운 issue를 자동으로 triage할 수 있습니다. 아래 예시는 스팸을 줄이기 위해 계정 생성 후 30일 이상인 사용자만 대상으로 필터링합니다.

issues event 역시 지시를 추출할 댓글이 없기 때문에 prompt 입력이 필수입니다.

Custom prompts

기본 prompt를 override해 워크플로에 맞게 SoryCode 동작을 커스터마이즈할 수 있습니다.

이 방식은 프로젝트별 리뷰 기준, 코딩 표준, 중점 점검 항목을 강제할 때 유용합니다.

예시

아래는 GitHub에서 SoryCode를 활용하는 대표 예시입니다.

  • Issue 설명 요청

    GitHub issue에 아래 댓글을 남기세요.

    SoryCode는 전체 스레드와 모든 댓글을 읽고 명확한 설명으로 답변합니다.

  • Issue 수정 요청

    GitHub issue에서 아래처럼 요청하세요.

    SoryCode가 새 branch를 만들고 변경을 구현한 뒤, 변경 사항이 담긴 PR을 생성합니다.

  • PR 리뷰 중 변경 요청

    GitHub PR에 아래 댓글을 남기세요.

    SoryCode가 요청한 변경을 구현하고 같은 PR에 커밋합니다.

  • 특정 코드 줄 리뷰 요청

    PR의 “Files” 탭에서 코드 줄에 직접 댓글을 남기세요. SoryCode는 파일, 줄 번호, diff 맥락을 자동으로 인식해 더 정확한 응답을 제공합니다.

    특정 줄 댓글에서는 SoryCode가 다음 정보를 함께 받습니다.

    • 검토 중인 정확한 파일
    • 해당 코드 줄
    • 주변 diff 맥락
    • 라인 번호 정보

    따라서 파일 경로나 라인 번호를 직접 적지 않아도 더 정밀하게 요청할 수 있습니다.
